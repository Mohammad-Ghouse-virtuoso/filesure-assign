## Question 1: Schema Design

In the ingestion pipeline, I normalized each CSV row into a flat MongoDB document with these main fields:

- `cin`
- `company_name`
- `status`
- `incorporation_date`
- `state`
- `directors`
- `paid_up_capital`
- `last_filing_date`
- `email`
- `email_valid`
- `data_quality`

I kept the schema mostly flat because the dataset is simple and the primary API queries are filter-based, not relationship-heavy.

The only array field is `directors`, because the source has `director_1` and `director_2`, and an array is cleaner than carrying multiple fixed columns forward.

I also kept `email_valid` as a separate boolean. That matters because the brief explicitly says to validate and flag invalid emails rather than dropping the record. So the schema preserves original data and still exposes quality issues.

For indexing, I added a compound index on `status` and `state`, because those are exactly the filter fields used by the API. That makes the chosen schema and the API shape consistent.

If this grew from 80 records to 10 lakh records, I would keep the same logical schema but change the operational approach:

- run against a real MongoDB instance instead of the fallback JSON path
- keep the compound filter index
- consider separate read optimization for summary queries
- move to cursor-based pagination if response volume became large
- add stronger request validation and maybe caching for the summary endpoint

## Question 2: Data Pipeline Decisions

In the Python script, the main focus was cleaning inconsistent data without crashing.

The CSV has multiple intentional quality problems, and I handled the main ones directly:

- inconsistent status values like `Active`, `active`, `Strike Off`, and `Under liquidation`
- mixed date formats like `15-03-2019` and `2019/03/15`
- dirty capital values with commas, currency markers, and odd characters
- blank CIN values
- blank director values
- invalid email formats

For status, I normalize to lowercase and map variants into a small set of standard values.

For dates, I use `dateutil` with `dayfirst=True`, which is safer than writing manual format branching for this kind of dataset.

For `paid_up_capital`, I remove formatting noise and convert to a numeric value.

For email handling, I deliberately do not drop bad rows. I preserve the email and set `email_valid` to `false`.

For missing fields, I use nullable values and defaults so the pipeline continues instead of crashing.

What I consciously did not over-engineer:

- I did not build a full validation report store outside the record itself.
- I did not add schema migration tooling because the dataset is small and the assignment is scoped.
- I did not reject rows that were incomplete but still useful. For example, rows with missing `cin` are kept if the company name exists.

## Question 3: API Structure

The API is in Express and exposes the required routes:

- `GET /companies`
- `GET /companies` with `status` and `state` filters
- `GET /companies/summary`

I also added `GET /health` and `GET /companies/stats` to support verification and the frontend.

The error handling is structured so that the API stays available even if MongoDB is not connected.

That mattered because the assignment says the API should return a clear error response or otherwise handle database issues without crashing.

In this implementation, if MongoDB is unavailable, the server falls back to `data.json`, which is generated by the ingestion script. That lets the frontend and API still function locally while the health route makes the active data source explicit.

I also tightened validation for `GET /companies`:

- unsupported query parameters now return a `400`
- invalid pagination values like `page=abc` return a `400`

That is better than silently accepting bad input, especially for support-facing and client-facing tooling.

On the query side, the `status + state` index supports the most important filtered lookup path. That keeps the schema and route behavior aligned.

## Question 4: What Broke

A few concrete things broke during implementation.

First, the Python ingestion script had static and runtime weaknesses around collection access and stats generation. I fixed that by adding explicit collection guards and correcting the aggregation-to-dictionary logic.

Second, the root route originally returned a 404 at `localhost:3000`. I fixed that by serving the frontend from `/`.

Third, the backend originally depended too heavily on MongoDB being present. Since the local environment did not have a working MongoDB instance, I added a fallback path from `data.json` so the API still works and still reports its real data source.

Fourth, the first version of the API was too permissive with bad query parameters. It accepted unsupported fields and malformed pagination instead of returning clear errors. I changed that and added integration tests to verify the corrected behavior.

I also hit a test startup delay because the MongoDB client timeout was too long when the database was offline. I reduced the server selection timeout so fallback mode activates quickly.

## Question 5: What I’d Change With One More Day

If I had one more day, I would improve four areas.

First, I would make the MongoDB path first-class instead of fallback-first in local testing, ideally with a reproducible local setup or container flow.

Second, I would add stronger automated coverage:

- more API tests
- ingestion unit tests for individual cleaners
- one lightweight frontend smoke test

Third, I would tighten the frontend slightly by surfacing API validation errors more clearly and possibly adding a simple search field.

Fourth, I would clean up the operational story for public sharing:

- keep `.env` local only
- publish `.env.example`
- add a better `.gitignore`
- wire the setup steps more tightly in the README
